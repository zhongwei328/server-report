六、飞书接入配置

飞书是 Hermes 原生支持的消息平台，接入后可用手机消息控制服务器。

准备飞书开放平台应用

  1. 登录 open.feishu.cn
  2. 「开发者后台」→「创建企业自建应用」
  3. 填写应用名称（例：Wei AI）、描述
  4. 左侧「凭证与基础信息」→ 记下 App ID 和 App Secret
  5. 左侧「事件订阅」→ 勾选「使用长连接接收事件」
  6. 左侧「权限管理」→ 申请以下权限：
    • im:message:send_as_bot（发消息）
    • im:message:receive_v1（收消息）
  7. 左侧「应用发布」→「创建版本」→「申请发布」

配置 Hermes 飞书 Bot

# 启动交互式配置
hermes gateway setup

# 选择 feishu，按提示填入：
# App ID: cli_xxxxxxxx
# App Secret: xxxxxxxxxxxxxxxx

# 启动网关
hermes gateway run
⚠️ 注意：服务器只有内网 IP（10.8.0.9），飞书长连接模式下不需要公网回调地址，直接用 WebSocket 与飞书服务器保持连接，穿透无压力。

八、多 Agent 架构：一体三分 vs. 三足鼎立

两种方案对比

方案 A：一个 Hermes + 三个子系统（Profile）

主 Agent
🧙 指挥官
理解任务 / 派发 / 汇总
→ 派发 →
子系统
🔍 研究员
搜索 / 查资料
子系统
✅ 质检员
复核 / 纠错

三个子系统在同一 Hermes 进程内，通过内部消息传递协作

方案 B：三个独立 Hermes Agent（多进程）

进程 1
🧙 指挥官
理解任务 / 派发 / 汇总
↔ 网络 / 文件 ↔
进程 2
🔍 研究员
搜索 / 查资料
进程 3
✅ 质检员
复核 / 纠错

三个独立进程，通过 tmux / 文件 / HTTP 互相通信

维度方案 A（一体三分）方案 B（三足鼎立）
隔离性差 — 一个崩溃全部崩好 — 进程独立，互不影响
防死机能力弱 — 主进程挂了就全挂强 — 任意一个可被其他救活
资源占用低 — 共享一个进程高 — 三个进程 × 内存
配置复杂度低 — 一个安装，一个配置高 — 三个配置，三个 Profile
通信延迟极低 — 内存内通信低 — 文件/网络，有毫秒延迟
并行能力受限 — 单进程并发有限强 — 真正并行，独立 CPU 核
长期稳定性风险集中容错冗余

防死机互救机制（方案 B 专用）

🔄 自动心跳检测 + 复活脚本

每个 Agent 每分钟向共享文件写入心跳时间戳。

独立的「守护进程」每分钟检查：

# 检查心跳，3分钟无跳动则判定为死机
if (( $(date +%s) - heartbeat_timestamp > 180 )); then
  tmux kill-session -t researcher
  sleep 2
  tmux new-session -d -s researcher "hermes -p researcher"
  echo "$(date +%s)" > ~/.hermes/heartbeat-researcher
fi

🔄 任务接替机制

如果研究员崩溃：指挥官检测到研究员输出文件缺失，自动重新派发任务给备用研究员（可以是同一进程的 delegate_task）。

如果指挥官崩溃：守护进程重启指挥官，指挥官从最近的会话快照恢复（Hermes 有会话保存机制）。

🏆 最终推荐：方案 B（三个独立 Hermes Agent）

你的服务器 2核/3.6G内存，跑三个 Hermes Agent 绑绑有余。

方案 A 适合「轻量快速实验」，方案 B 才能实现真正的容错互救——你提到的「防止一个Agent死机、可被救活」只有方案 B 能做到。

推荐配置：

  • 指挥官：hermes -p commander
  • 研究员：hermes -p researcher
  • 质检员：hermes -p qa

三者通过共享文件目录 ~/.hermes/shared/ 传递任务和结果。
