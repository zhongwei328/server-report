二、飞书接入配置

飞书是 Hermes 原生支持的消息平台，接入后可用手机消息控制服务器。内网服务器使用 WebSocket 长连接，无需公网 IP。

准备飞书开放平台应用

  1. 登录 open.feishu.cn → 开发者后台 → 创建企业自建应用
  2. 左侧「凭证与基础信息」→ 记下 App ID 和 App Secret
  3. 左侧「事件订阅」→ 勾选「使用长连接接收事件」
  4. 左侧「权限管理」→ 申请：im:message:send_as_bot、im:message:receive_v1
  5. 左侧「应用发布」→「创建版本」→「申请发布」

配置 Hermes

hermes gateway setup   # 选择 feishu，填入 App ID / Secret
hermes gateway run     # 启动网关
⚠️ 注意：腾讯云服务器只有内网 IP（10.8.0.9），飞书 WebSocket 长连接模式下不需要公网回调地址，直接与飞书服务器建立连接，穿透无压力。

三、WorkBuddy vs. Hermes：不是竞争，是互补

✅ 重要发现：WorkBuddy 和 Hermes 是两个完全不同定位的工具，并非竞争关系，一起使用效果更好。

🔵 WorkBuddy（腾讯出品）

定位
  • 本地桌面客户端（Windows / Mac）
  • 腾讯生态深度集成
  • 擅长本地文件操作
擅长
  • 操作腾讯文档、腾讯会议
  • 本地文件批量处理
  • 生成 PPT / 报告（本地）
  • 企业微信 / 微信助理接入
局限
  • 无法 24 小时在线（依赖电脑开机）
  • 无法连接服务器/云端工具
  • 无法定时任务
  • 没有多 Agent 协作框架
价格
  • 有免费额度，积分制

🟣 Hermes Agent

定位
  • 服务器端 AI Agent（云端运行）
  • 多平台消息接入
  • 多 Agent 协作框架
擅长
  • 24 小时在线（服务器常驻）
  • 定时任务 / cron 自动化
  • 网络搜索、网页读取
  • 多 Agent 协作（研究员+质检员）
  • 接飞书 / Telegram 等
局限
  • 无法直接操作腾讯文档（需 API）
  • 配置有一定门槛
价格
  • 免费开源，API 费用自付

两者如何协作？

WorkBuddy（本地电脑）          Hermes（腾讯云服务器）
┌─────────────────┐            ┌─────────────────────┐
│  操作腾讯文档    │ ←────────→ │  24小时监控 + 搜索   │
│  生成 PPT       │  数据文件   │  飞书消息收发        │
│  整理本地文件    │  任务指令   │  定时报告生成        │
│  企业微信助理    │            │  多Agent协作质检     │
└─────────────────┘            └─────────────────────┘

实际场景：Hermes 在服务器上完成调研和报告草稿 → 通过飞书发给你 → 你在 WorkBuddy 里润色、排版、发腾讯文档。两边各做擅长的事。

WorkBuddy 支持的消息平台接入

WorkBuddy 支持：微信助理（推荐）、微信客服号、企微助理、QQ 机器人、飞书、钉钉

飞书两者都支持，但定位不同：WorkBuddy 用飞书接收任务指令，Hermes 用飞书作为主控制通道。

维度方案 A（一体三分）方案 B（三足鼎立）⭐

防死机能力弱强 — 可被其他 Agent 救活
资源占用低中等（三进程）
配置复杂度低中等
并行能力受限真正并行
容错冗余风险集中有备份

防死机互救机制（方案 B）

🔄 心跳检测 + 自动复活

每个 Agent 每分钟向共享文件写入心跳时间戳。守护脚本检查：3分钟无心跳则判定死机，自动重启对应 tmux session。

# 检查研究员心跳
last=$(cat ~/.hermes/shared/heartbeat-researcher 2>/dev/null || echo 0)
now=$(date +%s)
if (( now - last > 180 )); then
  tmux kill-session -t researcher
  sleep 2
  tmux new-session -d -s researcher "hermes -p researcher"
  echo "$now" > ~/.hermes/shared/heartbeat-researcher
fi

🔄 任务接替

如果研究员崩溃：指挥官检测到研究员输出文件缺失，自动重新派发任务。
如果指挥官崩溃：守护进程重启指挥官，从最近会话快照恢复（Hermes 自动保存）。

🏆 推荐：方案 B（三个独立 Hermes Agent）

你的服务器 2核/3.6G内存跑三个 Hermes Agent 绑绑有余。

你提到的「防止一个Agent死机、可被救活」只有方案 B 能做到。

启动命令：

hermes -p commander   # 指挥官
hermes -p researcher   # 研究员
hermes -p qa           # 质检员

三者通过共享目录 ~/.hermes/shared/ 传递任务和结果。

五、下一步行动路线

1️⃣ 接入飞书

按第二节步骤配置，WebSocket 模式无需公网 IP。

2️⃣ 搭建三 Agent 互救系统

用 tmux 启动三个独立 Hermes Profile，配置心跳 + 自动复活。

3️⃣ WorkBuddy + Hermes 协作

WorkBuddy 负责腾讯文档本地操作，Hermes 负责服务器端任务和飞书消息。

4️⃣ 测试互救机制

手动 kill 掉一个 Agent，验证守护脚本能否在 3 分钟内自动复活。
